## Summary

This is an initial project commit that establishes the MCP (Model Context Protocol) infrastructure for Aion — a Flutter desktop astrology app. It introduces a plugin host, workspace store, manifest system, Riverpod providers, a plugin status UI page, documentation, and 18 passing tests (15 unit + 3 integration). The bulk of the 144 files are Flutter/platform scaffolding; the meaningful application code lives in `lib/mcp/`, `lib/providers/`, `lib/widgets/`, and `test/`.

Overall this is a well-structured foundation. The architectural choices (MCP-as-spine, sealed class state machine, BehaviorSubject streams, Riverpod bridge) are clean and appropriate. The findings below range from correctness bugs to hardening suggestions.

## Critical Findings

### CRIT-1: PlugHost error state can be orphaned on connection failure

**File:** `lib/mcp/plugin_host.dart:74-81`

```dart
} catch (e) {
  _stateOf(manifest.name).add(PluginState.error(e));
}
```

If `client.connect(transport)` succeeds but `client.listTools()` throws, the `catch` block sets error state, but the client is never cleaned up. The `_clients` map retains a stale `McpClient` entry for `manifest.name`. Subsequent `callTool()` calls will hit `_clients[server]` and find a disconnected client rather than raising `PluginNotConnected`.

A `stopPlugin` in the catch, or at minimum removing the client from `_clients` on failure, would prevent this leak. Consider:

```dart
} catch (e) {
  _clients.remove(manifest.name);
  _stateOf(manifest.name).add(PluginState.error(e));
}
```

### CRIT-2: BehaviorSubject instances leaked on stop

**File:** `lib/mcp/plugin_host.dart:93-97`

```dart
Future<void> stopPlugin(String name) async {
  final client = _clients.remove(name);
  await client?.close();
  _stateOf(name).add(const PluginState.stopped());
}
```

`stopPlugin` pushes `stopped` onto the `BehaviorSubject` but never closes it. `dispose()` iterates `_clients.keys` calling `stopPlugin`, but `_states` entries are never closed. Compare with `WorkspaceStore.dispose()`, which correctly closes its subjects. This means `PluginHost.dispose()` leaks every `BehaviorSubject` it created.

### CRIT-3: `startAll` silently swallows all errors

**File:** `lib/mcp/plugin_host.dart:99-106`

```dart
Future<void> startAll(List<PluginManifest> manifests) async {
  await Future.wait(
    manifests.where((m) => m.autoStart).map((m) async {
      try {
        await startPlugin(m);
      } catch (_) {}
    }),
  );
}
```

Every exception is caught and discarded. The inner `startPlugin` already sets `PluginState.error`, so the outer try-catch is redundant for state tracking, but it also means any unexpected error type is silently dropped. If callers need to know which plugins failed, this information is lost to the void. At minimum, log the errors; better, return a structured result.

## High-Severity Findings

### HIGH-1: No timeout on MCP client connect or tool calls

**File:** `lib/mcp/plugin_host.dart:68-80`

`client.connect(transport)` and `client.callTool(...)` have no timeout. If a plugin process hangs during startup or a tool call stalls, the entire app UI freezes waiting on the future. The handoff document acknowledges this as acceptable for Phase 1, but it is a real reliability risk. `startPlugin` should use `.timeout(Duration(seconds: N))` and `callTool` likewise.

### HIGH-2: No process exit / crash detection

If the spawned MCP server process exits (crash, OOM, etc.), the `StdioClientTransport` stream will close, but `PluginHost` never monitors for this. The state remains `connected` until the next `callTool` fails. Consider subscribing to the transport's close event or adding a periodic health-check ping to update state to `error` promptly.

### HIGH-3: `PluginConfig.loadUserPlugins` has no error handling for malformed JSON

**File:** `lib/mcp/plugin_manifest.dart:99-107`

```dart
static Future<List<PluginManifest>> loadUserPlugins() async {
  final file = File(_pluginsPath());
  if (!await file.exists()) return [];
  final contents = await file.readAsString();
  final jsonList = jsonDecode(contents) as List;
  return jsonList.map((e) => PluginManifest.fromJson(e as Map<String, dynamic>)).toList();
}
```

If the file contains invalid JSON or the wrong structure (e.g. a map instead of a list), this crashes with an unhandled exception. A corrupt `plugins.json` could prevent the app from launching. Wrap in try-catch and return an empty list or a structured error.

### HIGH-4: `SlotReady.data` and `SlotError.error` use `dynamic` map values

**File:** `lib/mcp/slot_state.dart:15-23`

```dart
class SlotReady extends SlotState {
  final Map<String, dynamic> data;
  final Map<String, dynamic> options;
  ...
}
class SlotError extends SlotState {
  final Object error;
  final Map<String, dynamic> options;
  ...
}
```

`Map<String, dynamic>` discards type information at the boundary. Downstream consumers must cast blindly. Consider a typed model or at minimum a typed alias for the slot data map. This is a design debt that compounds as more tools are added.

## Medium-Severity Findings

### MED-1: `BehaviorSubject.seeded` vs `BehaviorSubject.seeded` inconsistency

`PluginHost._stateOf` uses `BehaviorSubject<PluginState>.seeded(...)` (line 30) while `WorkspaceStore._slotOf` uses `BehaviorSubject.seeded(...)` (workspace_store.dart:14). The `.seeded` named constructor works the same way in both cases (rxdart accepts it), but the explicit generic in `PluginHost` is preferred. Be consistent — use `BehaviorSubject<SlotState>.seeded(...)` in `WorkspaceStore` too.

### MED-2: `_substituteVars` only handles `${AION_PLUGINS}`

**File:** `lib/mcp/plugin_manifest.dart:75-78`

```dart
static String _substituteVars(String value) {
  final pluginsDir = Platform.environment['AION_PLUGINS'] ?? '${configPath()}/plugins';
  return value.replaceAll(r'${AION_PLUGINS}', pluginsDir);
}
```

Only one variable is supported. The naming `_substituteVars` (plural) and the `$` + `{}` syntax suggest extensibility, but there's no mechanism to add more variables. Either rename to `_substituteAionPlugins` or implement a general `${VAR}` resolver that pulls from `Platform.environment` plus app-specific overrides.

### MED-3: Test mock extends production class with single-field override

**File:** `test/mcp/workspace_store_test.dart:8-18`

```dart
class MockPluginHost extends PluginHost {
  CallToolResult? nextResult;
  Object? nextError;

  @override
  Future<CallToolResult> callTool(...) async {
    if (nextError != null) throw nextError!;
    return nextResult!;
  }
}
```

This mock inherits all of `PluginHost`'s state (BehaviorSubjects, client map) but only overrides `callTool`. If `MockPluginHost` is ever used where `startPlugin`/`stopPlugin` are called, the real BehaviorSubject map interferes. Consider using a proper mocking framework (e.g. `mocktail`) or extracting an interface.

### MED-4: `PluginManifest.fromJson` doesn't validate required fields

**File:** `lib/mcp/plugin_manifest.dart:34-57`

If `name`, `displayName`, or `description` are missing from the JSON, the `as String` cast will throw a generic `TypeError`. A structured validation error (e.g. `FormatException('Missing required field: name')`) would be more helpful for plugin developers debugging their manifest files.

### MED-5: Integration test `@Tags(['integration'])` is not wired to test runner

**File:** `test/integration/drishti_test.dart:1`

`@Tags(['integration'])` requires `package:test` tag configuration to actually filter. Without a `dart_test.yaml` or `test/settings.yaml` defining the tag, running `flutter test` will execute integration tests that spawn a real Drishti process. This can cause CI failures if Drishti isn't available. Add tag configuration or use `--exclude-tags integration` in CI.

### MED-6: `PluginStatusPage` hardcodes `BundledManifests.drishti`

**File:** `lib/widgets/plugin_status.dart:24-26`

```dart
Future.microtask(() {
  ref.read(pluginHostProvider).startAll([BundledManifests.drishti]);
});
```

and line 36:

```dart
children: [BundledManifests.drishti].map(_buildPluginTile).toList(),
```

Both the startup list and the rendering list are hardcoded to a single manifest. Phase 1 is clear that only Drishti is bundled, but this should at minimum reference a list (`BundledManifests.all`) to avoid duplication and ease the addition of new plugins.

### MED-7: `workspaceStoreProvider` uses `ref.read` instead of `ref.watch`

**File:** `lib/providers/workspace_providers.dart:12`

```dart
final store = WorkspaceStore(ref.read(pluginHostProvider));
```

`ref.read` is correct here since the host doesn't change and we don't want the store rebuilt on host changes. However, if the host is ever replaced (disposed and recreated), the store holds a stale reference. This is fine for Phase 1 but worth a comment documenting the intentional choice.

### LOW-2: `PluginHost` lacks logging

No logging whatsoever in `PluginHost`, `WorkspaceStore`, or `PluginConfig`. Adding `dart:developer` log statements at key lifecycle points (connect, disconnect, error) would significantly ease debugging of plugin startup issues in production.

### LOW-3: `SlotIdle`, `SlotLoading`, `SlotReady`, `SlotError` are in `slot_state.dart` but are also used in tests and widgets

The sealed class is small and clean, but the `options` field on `SlotLoading` is `Map<String, dynamic>` which may not carry meaningful data at load time. Consider whether `SlotLoading` really needs an options map or if it's just for symmetry.

### LOW-4: `connectedPlugins` getter iterates all entries

**File:** `lib/mcp/plugin_host.dart:108-111`

```dart
List<String> get connectedPlugins => _states.entries
    .where((e) => e.value.value.status == PluginStatus.connected)
    .map((e) => e.key)
    .toList();
```

This creates a new list on every call. Fine for current scale, but if queried frequently, consider caching or exposing as a stream.

### LOW-5: `PluginManifest.toJson()` omits null fields with `if` guards

This is actually fine — it means round-tripping `toJson()` → `fromJson()` preserves null fields as absent rather than explicit nulls. Just note that `fromJson(jsonDecode(jsonEncode(manifest)))` will not preserve the absence of optional fields identically if someone relies on `containsKey` checks.

### LOW-6: `analysis_options.yaml` uses `flutter_lints` (deprecated)

```yaml
include: package:flutter_lints/flutter.yaml
```

Flutter's recommended lint set has been renamed to `flutter_lints` → the newer convention is `package:flutter_lints/flutter.yaml` which is what's used here. However, as of Dart 3.x, the canonical package is actually `flutter_lints` which has been succeeded by the `lints` package coupled with `flutter_lints`. Just verify this resolves correctly with `flutter analyze`.

## Architecture Observations

1. **Clean separation of concerns.** `lib/mcp/` is pure Dart with no Flutter dependency — testable in isolation. The Riverpod bridge in `lib/providers/` is thin and mechanical. Good.

2. **Sealed class state machine** (`SlotState`, `PluginState`) is idiomatic Dart 3 and forces exhaustive matching. Good choice.

3. **BehaviorSubject for state streams** is appropriate for the "latest state + stream" pattern. Ensure disposal is thorough (see CRIT-2).

4. **`startAll` uses `Future.wait`** meaning all plugins start concurrently. For Phase 1 this is fine, but for ordering dependencies between plugins you'll want sequential or DAG-based startup.

5. **No dependency injection.** `PluginHost` directly instantiates `McpClient`. For testing beyond the simple `MockPluginHost`, consider accepting a `McpClientFactory` function or using a provider-based factory.

6. **The doc files** (`architecture-overview.md`, `roadmap.md`, `handoff.md`) are thorough and track open questions. This is a strong practice.

## Summary Verdict

The MCP spine foundation is solid. The two critical issues are the client leak on partial connection failure (CRIT-1) and the BehaviorSubject leak in `PluginHost.dispose()` (CRIT-2). Both are simple fixes that should go in before Phase 2. The error-swallowing in `startAll` (CRIT-3) is concerning but not blocking. The architectural debt items (DI, timeouts, crash detection) are correctly deferred in the handoff doc but should be tracked as hardening tasks.
