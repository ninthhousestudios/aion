## Executive Summary

This commit introduces the foundational MCP spine for Aion, a professional astrology desktop application. The implementation is **solid for a Phase 1 milestone** ‚Äî clean separation of concerns, good test coverage (18 tests, all passing), and pragmatic choices around the `mcp_dart` fork. However, there are several categories of issues that should be addressed before Phase 2: **resource leaks**, **missing error handling**, **async/await anti-patterns**, **analyzer warnings**, and **architectural debt** that will compound if left unchecked.

**Overall Grade: B+** ‚Äî Good foundation, needs hardening before scaling.

---

## Architecture Assessment

### Strengths

1. **Clean Layering** ‚Äî The separation between `lib/mcp/` (pure Dart business logic), `lib/providers/` (Riverpod bridge), and `lib/widgets/` (UI) is well-executed. This will make testing and porting to other state management solutions straightforward.

2. **MCP as Spine** ‚Äî The architecture document (`doc/architecture-overview.md`) clearly articulates the MCP-first design. The code delivers on this vision: `PluginHost` manages server lifecycles, `WorkspaceStore` abstracts slot state, and the UI is indeed a thin shell.

3. **Riverpod Integration** ‚Äî Using `Provider` for `PluginHost` and `WorkspaceStore` (long-lived singletons) and `StreamProvider.family` for per-plugin/per-slot state is idiomatic Riverpod. The `ref.onDispose()` hooks are correctly wired.

4. **Sealed Class for SlotState** ‚Äî `SlotState` as a sealed class with `SlotIdle`, `SlotLoading`, `SlotReady`, and `SlotError` is a modern, type-safe Dart pattern that eliminates invalid state combinations.

5. **Test Coverage** ‚Äî 15 unit tests + 3 integration tests, all green. The mock-based unit tests for `WorkspaceStore` and the real-process integration tests for Drishti provide good confidence.

### Concerns

1. **BehaviorSubject Leakage** ‚Äî `PluginHost` and `WorkspaceStore` both expose `BehaviorSubject` directly via `watchPlugin()` and `watch()`. Callers get a raw `Stream`, which is good, but the internal management of these subjects is fragile (see Resource Leaks below).

2. **PluginHost is a God Object in Training** ‚Äî It manages transports, clients, state streams, and tool calls. At 137 lines it's still manageable, but as more transport types and lifecycle features are added, it will grow. Consider extracting `TransportFactory` and `PluginLifecycleManager` in Phase 2.

3. **No Timeout or Crash Detection** ‚Äî Acknowledged in `handoff.md`, but worth emphasizing: `McpClient.connect()` and `callTool()` are unbounded. If Drishti hangs, the UI hangs. If Drishti crashes, `PluginHost` doesn't know until the next `callTool` fails.

## Critical Issues

### 1. Resource Leak: BehaviorSubjects Never Removed (HIGH)

**File:** `lib/mcp/plugin_host.dart` (lines 52-60), `lib/mcp/workspace_store.dart` (lines 11-17)

```dart
// PluginHost
final _states = <String, BehaviorSubject<PluginState>>{};

BehaviorSubject<PluginState> _stateOf(String name) {
  return _states.putIfAbsent(
    name,
    () => BehaviorSubject<PluginState>.seeded(const PluginState.stopped()),
  );
}
```

When `stopPlugin()` is called, the `BehaviorSubject` is NOT removed from `_states` ‚Äî only the client is closed and a new `PluginState.stopped()` is emitted. Similarly, `WorkspaceStore.dispose()` closes subjects but doesn't clear the `_slots` map. Over repeated start/stop cycles, this leaks memory and accumulated stream listeners.

**Fix:** Remove the subject from the map on stop/close:
```dart
Future<void> stopPlugin(String name) async {
  final client = _clients.remove(name);
  await client?.close();
  final subject = _states.remove(name);  // <-- remove
  subject?.close();                       // <-- close and discard
}
```

### 2. PluginHost.dispose() Does Not Clear Internal Maps (HIGH)

**File:** `lib/mcp/plugin_host.dart` (lines 128-131)

```dart
Future<void> dispose() async {
  final names = List<String>.from(_clients.keys);
  await Future.wait(names.map(stopPlugin));
}
```

After `dispose()`, if `stopPlugin` is fixed to `_states.remove(name)`, `_clients` and `_states` will be empty. But as written, even after fixing `stopPlugin`, `_states` may still contain entries for plugins that were never started (since `_stateOf` eagerly creates subjects). Also, `dispose()` does not prevent subsequent `startPlugin` calls ‚Äî the object is in a zombie state.

**Fix:** Add a `_disposed` flag and guard all public methods:
```dart
bool _disposed = false;

Future<void> dispose() async {
  if (_disposed) return;
  _disposed = true;
  // ... existing cleanup ...
  _clients.clear();
  _states.clear();
}
```

### 3. PluginManifest.fromJson Crashes on Unknown Transport (MEDIUM)

**File:** `lib/mcp/plugin_manifest.dart` (lines 34-35)

```dart
final transport = transportStr == 'http' ? PluginTransport.http : PluginTransport.stdio;
```

Any value other than `'http'` silently falls back to `stdio`. A typo like `'stio'` or `'websocket'` would not be caught. This violates the fail-fast principle.

**Fix:**
```dart
final transport = switch (transportStr) {
  'stdio' => PluginTransport.stdio,
  'http' => PluginTransport.http,
  _ => throw FormatException('Unknown transport: $transportStr'),
};
```

### 4. PluginConfig._substituteVars is Not Tested for Missing HOME (MEDIUM)

**File:** `lib/mcp/plugin_manifest.dart` (lines 83-95)

```dart
static String configPath() {
  final xdg = Platform.environment['XDG_CONFIG_HOME'];
  final base = xdg ?? '${Platform.environment['HOME']}/.config';
  return '$base/aion';
}
```

If both `XDG_CONFIG_HOME` and `HOME` are missing (e.g., in a restricted sandbox or CI), `configPath()` returns `null/.config/aion`, which will crash later when creating the directory. The `testVariableSubstitution` test doesn't exercise this edge case.

**Fix:** Add a fallback: `Platform.environment['HOME'] ?? Platform.environment['USERPROFILE'] ?? '.'`

### 5. WorkspaceStore.recalculate Parses JSON Without Checking isError (MEDIUM)

**File:** `lib/mcp/workspace_store.dart` (lines 23-48)

```dart
final result = await _host.callTool(server, tool, args);
final textContent = result.content.whereType<TextContent>().firstOrNull;
```

`CallToolResult` has an `isError` flag. If the MCP server returns an error (e.g., invalid chart parameters), `isError` is true but `content` may still contain `TextContent` with an error message. The current code blindly tries to JSON-decode it, leading to a confusing `SlotError('Tool result is not valid JSON: ...')` instead of surfacing the actual server error.

**Fix:**
```dart
if (result.isError) {
  final message = result.content.whereType<TextContent>().firstOrNull?.text ?? 'Unknown tool error';
  subject.add(SlotError(message, args));
  return;
}
```

### 6. PluginStatusPage initState Starts Plugins Without Await (MEDIUM)

**File:** `lib/widgets/plugin_status.dart` (lines 18-24)

```dart
@override
void initState() {
  super.initState();
  Future.microtask(() {
    ref.read(pluginHostProvider).startAll([BundledManifests.drishti]);
  });
}
```

`startAll()` is fire-and-forget. If it throws synchronously (unlikely but possible), the error is swallowed by the microtask. More importantly, there's no loading state management ‚Äî the widget relies on the stream to show progress, which is fine, but there's no retry or error surface for initialization failures.

**Recommendation:** At minimum, catch and log:
```dart
Future.microtask(() async {
  try {
    await ref.read(pluginHostProvider).startAll([BundledManifests.drishti]);
  } catch (e, st) {
    debugPrint('Failed to start plugins: $e\n$st');
  }
});
```

### 7. callTool Throws on Disconnected Server But Error is Non-Actionable (LOW)

**File:** `lib/mcp/plugin_host.dart` (lines 98-108)

```dart
Future<CallToolResult> callTool(...) async {
  final client = _clients[server];
  if (client == null) {
    throw PluginNotConnected(server);
  }
  return client.callTool(...);
}
```

`PluginNotConnected` is a good custom exception, but callers (`WorkspaceStore.recalculate`) catch it as a generic `catch (error)` and wrap it in `SlotError`. The UI then displays `Exception: PluginNotConnected: drishti` which is not user-friendly.

**Recommendation:** Consider adding a user-facing message field to `PluginNotConnected`.

## Code Quality Issues

### Analyzer Warnings (2 issues)

1. **`lib/mcp/plugin_host.dart:39`** ‚Äî `prefer_initializing_formals`  
   `PluginState.error(Object error)` should use `this.error` or rename the parameter to avoid shadowing.

2. **`test/integration/drishti_test.dart:1`** ‚Äî `library_annotations`  
   `@Tags(['integration'])` should be attached to a `library` directive, not a standalone annotation. Add `library drishti_test;` at the top, or create `dart_test.yaml` to define the `integration` tag.

   **Fix:** Create `dart_test.yaml`:
   ```yaml
   tags:
     integration:
   ```

### Style & Idioms

1. **Missing `const` constructors** ‚Äî `PluginHost`, `WorkspaceStore`, and `PluginConfig` could benefit from `const` constructors where possible. Not critical, but improves performance and immutability signaling.

2. **`PluginManifest` equality** ‚Äî No `==` or `hashCode` overrides. Two manifests with identical fields are not equal. This can cause unnecessary rebuilds in Riverpod if manifests are used as keys or in `select` comparisons.

3. **`toJson()` does not round-trip `url` field correctly for bundled manifests** ‚Äî The bundled `drishti` manifest has `url: null`, which `toJson()` omits. That's fine. But if a user manifest has `url: ''` (empty string), it would be serialized and then deserialized as `url: ''`, which is semantically different from `url: null` in some contexts. Consider validating URL non-emptiness.

4. **No `copyWith` on data classes** ‚Äî `PluginManifest`, `PluginState`, and `SlotState` lack `copyWith` methods. These are heavily used in UI and state management; manual reconstruction is error-prone.

5. **Hard-coded paths in bundled manifest** ‚Äî `BundledManifests.drishti` uses `../arjuna` as the default working directory. This is a fragile relative path that depends on the app's execution context. It works for development but will break in production builds. Document this clearly or make it configurable at build time.

## Testing Assessment

### Strengths

- **Good unit test coverage** for `PluginManifest` (parsing, HTTP transport, variable substitution, round-tripping), `PluginHost` (start, call, stop, error, stream), and `WorkspaceStore` (lifecycle, error, JSON, independent slots, active slots).
- **Integration tests** actually spawn Drishti and verify real tool calls. This is the right choice for MCP infrastructure.
- **Mock-based tests** for `WorkspaceStore` isolate from `PluginHost` correctly.

### Weaknesses

1. **Missing `dart_test.yaml`** ‚Äî The integration tag warning is a real issue. It means `flutter test` without the tag still runs integration tests, and `flutter test --tags integration` doesn't work as expected.

2. **PluginHost stream test is brittle** ‚Äî `testPluginStateStream` uses `await Future<void>.delayed(Duration.zero)` to flush the microtask queue. This is a known workaround (documented in `handoff.md`) but it's inherently flaky. A better approach is to use `expectLater` with `emitsInOrder`:
   ```dart
   expectLater(
     host.watchPlugin('test-server').map((s) => s.status),
     emitsInOrder([PluginStatus.stopped, PluginStatus.starting, PluginStatus.connected]),
   );
   await host.startPlugin(_testManifest);
   ```

3. **No test for `startAll` error swallowing** ‚Äî `startAll` has `catch (_) {}` which silently ignores failures. There is no test verifying that one plugin failing doesn't prevent others from starting. More importantly, there's no test asserting that errors ARE swallowed (which is the current behavior).

4. **No test for `PluginHost.dispose()`** ‚Äî The most critical cleanup path is untested. If `dispose()` is called while plugins are starting, it may deadlock or throw.

5. **WorkspaceStore tests don't verify disposal** ‚Äî `tearDown` calls `store.dispose()`, but there's no assertion that streams are closed or that `watch()` throws/hangs after disposal.

6. **Integration tests depend on external workspace** ‚Äî The Drishti tests require `../arjuna` to exist. This is fine for local dev but will fail in CI. Consider skipping gracefully:
   ```dart
   setUpAll(() {
     final drishtiPath = Platform.environment['DRISHTI_PATH'] ?? '../arjuna';
     final dir = Directory(drishtiPath);
     if (!dir.existsSync()) {
       skip('Drishti workspace not found at $drishtiPath');
     }
   });
   ```
   (Note: `skip` should be inside the test or `setUpAll` should throw a `SkipException` if your test runner supports it. The current `throw StateError` is a hard failure.)

## Security & Safety

1. **Arbitrary Command Execution** ‚Äî `PluginManifest.command` can be any executable. `PluginConfig.loadUserPlugins()` reads from `~/.config/aion/plugins.json`. If an attacker can write to that file, they can execute arbitrary commands. This is standard for plugin architectures (VS Code, Neovim, etc.), but:
   - Document the security model.
   - Consider validating commands against an allowlist or requiring user confirmation for non-bundled plugins.
   - Ensure `plugins.json` has restrictive file permissions (not created by default).

2. **Environment Variable Leakage** ‚Äî `PluginManifest.env` is passed directly to the child process. There's no filtering of sensitive env vars (like `PATH`, `HOME`, or API keys). Again, standard behavior, but worth documenting.

3. **JSON Parsing Without Schema Validation** ‚Äî `PluginManifest.fromJson` assumes the JSON shape is correct. Missing required fields will throw `TypeError` (e.g., `json['name'] as String` when `name` is missing). Consider using `json_serializable` or `freezed` for generated, type-safe serialization.

## Recommendations (Prioritized)

### Before Phase 2 (Must Do)

1. **Fix resource leaks in `PluginHost` and `WorkspaceStore`** ‚Äî Remove/clear subjects on stop/dispose.
2. **Add `_disposed` guard to `PluginHost`** ‚Äî Prevent use-after-dispose.
3. **Fix analyzer warnings** ‚Äî `prefer_initializing_formals` and `library_annotations`.
4. **Create `dart_test.yaml`** ‚Äî Define the `integration` tag properly.
5. **Handle `result.isError` in `WorkspaceStore.recalculate`** ‚Äî Distinguish MCP tool errors from JSON parse errors.
6. **Add strict transport validation in `PluginManifest.fromJson`**.
7. **Add `skip` or graceful failure to Drishti integration tests** when workspace is missing.

### Phase 2 Hardening (Should Do)

8. **Add timeouts to `McpClient.connect()` and `callTool()`** ‚Äî Acknowledged in `handoff.md`.
9. **Add process exit detection** ‚Äî Listen to `Process.exitCode` or transport close events.
10. **Add `copyWith`, `==`, `hashCode` to data classes** ‚Äî Use `freezed` or `equatable`.
11. **Document public APIs with DartDoc**.
12. **Replace `Future.delayed(Duration.zero)` flushes with `expectLater`/`pumpEventQueue`** in tests.
13. **Add test for `PluginHost.dispose()` during active connections**.
14. **Validate `PluginConfig.configPath()` when `HOME` is unset**.

### Long Term (Nice to Have)

15. **Refactor `PluginHost`** ‚Äî Extract `TransportFactory` and `PluginLifecycleManager`.
16. **Replace manual JSON with `json_serializable` or `freezed`**.
17. **Add a `PluginRegistry` abstraction** ‚Äî Instead of `PluginHost` knowing about bundled vs user plugins, delegate to a registry that loads from manifest files.
18. **Security model documentation** ‚Äî Plugin sandboxing, command allowlists, etc.

## Conclusion

This commit successfully delivers the Phase 1 MCP spine with a solid architectural foundation. The code is readable, well-layered, and adequately tested. The main risks are **resource management** (leaking BehaviorSubjects), **error handling gaps** (ignoring `isError`, weak validation), and **process lifecycle robustness** (no timeout, no crash detection).

These issues are all fixable and well within the scope of pre-Phase-2 hardening. Addressing the "Before Phase 2" recommendations will elevate the codebase to an A-grade foundation for the canvas and renderer work ahead.
